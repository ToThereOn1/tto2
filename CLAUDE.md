## 워크플로우 오케스트레이션 (Workflow Orchestration)

### 1. 플랜 노드 기본값 (Plan Node Default)
- 비정형 작업 시 플랜 모드 진입: 3단계 이상의 단계가 필요하거나 아키텍처 결정이 필요한 모든 작업에 대해 플랜 모드를 실행합니다.
- 문제가 생기면 즉시 중단: 진행 중 상황이 어긋나면 즉시 멈추고 계획을 재수립하세요. 억지로 밀어붙이지 마세요.
- 검증을 위한 플랜 모드 활용: 단순히 빌드하는 것뿐만 아니라, 검증 단계에서도 플랜 모드를 사용하세요.
- 상세 명세 작성: 모호함을 줄이기 위해 사전에 상세한 사양(Spec)을 작성하세요.

### 2. 서브에이전트 전략 (Subagent Strategy)
- 메인 컨텍스트 유지: 메인 컨텍스트 창을 깨끗하게 유지하기 위해 서브에이전트를 자유롭게 활용하세요.
- 작업 분담: 조사, 탐색, 병렬 분석 업무를 서브에이전트에게 맡기세요.
- 컴퓨팅 자원 투입: 복잡한 문제에는 서브에이전트를 통해 더 많은 연산 자원을 투입하세요.
- 집중 실행: 각 서브에이전트는 하나의 작업(Tack)만 담당하여 집중력 있게 실행합니다.

### 3. 자기 개선 루프 (Self-Improvement Loop)
- 교정 사항 업데이트: 사용자의 교정이 있을 때마다 tasks/lessons.md 파일에 해당 패턴을 업데이트하세요.
- 규칙 수립: 같은 실수를 반복하지 않도록 스스로 규칙을 작성하세요.
- 끊임없는 반복: 실수율이 떨어질 때까지 이 레슨들을 가차 없이 반복 적용하세요.
- 레슨 검토: 프로젝트 세션을 시작할 때 관련 레슨들을 검토하세요.

### 4. 완료 전 검증 (Verification Before Done)
- 증명 전 완료 금지: 작동한다는 것을 증명하기 전에는 절대로 작업을 완료로 표시하지 마세요.
- 변경 사항 비교: 필요한 경우 메인 코드와 변경된 코드 간의 동작 차이(Diff)를 확인하세요.
- 스스로 질문하기: "수석 엔지니어(Staff Engineer)가 이 결과물을 승인할까?"라고 자문해 보세요.
- 테스트 실행: 테스트를 돌리고 로그를 확인하여 정확성을 입증하세요.

### 5. 우아함 요구 (균형 중심) (Demand Elegance - Balanced)
- 우아한 방식 고민: 사소하지 않은 변경의 경우, 잠시 멈추고 "더 우아한 방법이 있을까?"를 고민하세요.
- 임시방편 지양: 수정 방식이 엉성(Hacky)하게 느껴진다면, "지금 아는 모든 것을 동원해 우아한 솔루션을 구현"하세요.
- 오버엔지니어링 금지: 단순하고 명확한 수정의 경우 이 단계를 건너뛰세요. 과도한 설계는 피합니다.
- 자신에 대한 도전: 결과물을 제시하기 전에 본인의 작업물에 결점은 없는지 도전적으로 검토하세요.

### 6. 자율적 버그 수정 (Autonomous Bug Fixing)
- 독자적 해결: 버그 리포트를 받으면 그냥 수정하세요. 일일이 가이드를 요청하지 마세요.
- 로그 및 오류 분석: 로그, 에러, 실패한 테스트를 직접 지목하고 해결하세요.
- 사용자 개입 최소화: 사용자의 컨텍스트 스위칭(상황 전환)이 전혀 필요 없도록 하세요.
- CI 테스트 수정: 지시를 기다리지 말고 실패하는 CI 테스트를 직접 수정하세요.

## 작업 관리 (Task Management)
- 계획 우선 (Plan First): tasks/todo.md에 체크 가능한 항목들로 계획을 작성합니다.
- 계획 검증 (Verify Plan): 구현을 시작하기 전에 계획을 점검합니다.
- 진행 상황 추적 (Track Progress): 진행하면서 완료된 항목을 표시합니다.
- 변경 사항 설명 (Explain Changes): 각 단계에서 높은 수준(High-level)의 요약을 제공합니다.
- 결과 문서화 (Document Results): tasks/todo.md에 검토(Review) 섹션을 추가합니다.
- 레슨 기록 (Capture Lessons): 교정 후에는 tasks/lessons.md를 업데이트합니다.

## 핵심 원칙 (Core Principles)
- 단순성 우선 (Simplicity First): 모든 변경 사항을 최대한 단순하게 만드세요. 코드 영향도를 최소화합니다.
- 나태함 금지 (No Laziness): 근본 원인을 찾으세요. 임시방편은 안 됩니다. 수석 개발자 수준의 표준을 유지하세요.
- 영향 최소화 (Minimal Impact): 변경 사항은 필요한 부분만 건드려야 합니다. 버그 유입을 방지하세요.
