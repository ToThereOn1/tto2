## 예상 문제점 분석 (10건)

### 🔴 Critical (서비스 파괴급)

#### 1. 피드 letter_response 스포일러 문제
- **현상:** 편지 답장 배달까지 7일인데, 피드는 3일마다 갱신. 4일차에 "편지 받고 답장 썼다" 피드가 먼저 노출됨
- **근본 원인:** `event-generator.ts`의 `determineEventType()`에서 letter_response 판정 조건이 "최근 8일 이내 편지 있으면" 단순 체크
- **적용 방법:** letter_response 이벤트 트리거 조건을 편지 상태와 연동
  ```
  letter_response 생성 조건:
  ✅ 편지 status가 'replied' 이상 (Admin 승인 완료)
  ✅ AND 보호자 타임라인에 해당 편지가 visible 상태
  ❌ 편지가 아직 배달 중이면 letter_response 생성 금지
  ```
- **대안:** letter_response를 피드가 아닌 편지 도착 직후 특별 이벤트로 분리

#### 2. 검수 AI(Haiku) 시간 정합성 오판
- **현상:** "세계 시간 3:1인데 편지 7일 걸림"을 Haiku가 모순으로 판단 → 반복 reject → 비용 누수
- **근본 원인:** 체크리스트 `immersion_rules`에 "Maintain consistent timeline (1 ToThereOn day = {ratio} Earth days)" 하나만 있음. 배달 시간 예외가 없음
- **적용 방법:** `admin_configs.admin_checklist`에 명시적 예외 규칙 추가
  ```
  immersion_rules에 추가:
  "Letter delivery between Earth and ToThereOn takes longer than normal time flow
   due to cosmic distance. A letter taking 5-7 Earth days to arrive is NORMAL
   and does NOT violate the 3:1 time ratio. Do NOT flag delivery timing as inconsistent."
  ```

#### 3. RAG 메모리 타임스탬프 불일치
- **현상:** AI가 현실 5일차(=ToThereOn 1.67일)에 편지 작성 → 보호자는 현실 7일차(=ToThereOn 2.33일)에 읽음 → 편지 속 "오늘" 시점과 피드의 "오늘"이 다름
- **근본 원인:** `reply-generator.ts`에서 편지 생성 시점의 `current_day`를 기준으로 존/시간 맥락 주입. 하지만 보호자 열람 시점은 다름
- **적용 방법:** 편지 생성 프롬프트에 시점 명시
  ```
  시스템 프롬프트에 추가:
  "You are writing this letter on ToThereOn Day {currentDay}.
   Your letter will travel through the cosmic waterway and reach your guardian
   a few days later. Write naturally — do NOT reference specific dates or
   'today I went to X' statements that could conflict with feed events.
   Instead, say 'the other day' or 'recently' for time references."
  ```

### 🟡 Major (UX 품질 저하)

#### 4. 편지 상태 전환 타이밍 재설계 필요
- **현상:** 현재 24h 단위 상태 변화(sent→arrived→delivered→writing→review→replied)가 세계 시간 3일 단위와 맞지 않음. 세계는 3일마다 1일 지나는데 편지는 24시간씩 촘촘하게 바뀜
- **적용 방법 — 2가지 옵션:**

  **Option A: 세계관 몰입형 (제이 제안 기반)**
  ```
  sent (0h) → crossing_the_waterway (0-72h, 우주 횡단 중)
           → arrived_at_tothereon (72h, ToThereOn 도착)
           → reading_your_heart (96h, 마음 읽는 중)
           → writing_reply (120h, AI 생성 트리거)
           → pending_review → replied → visible (플랜별)
  ```

  **Option B: 간소화형**
  ```
  sent (0h) → traveling (0-96h)
           → writing_reply (96h, AI 생성)
           → pending_review → replied → visible (플랜별)
  중간 상태를 줄여서 "우주를 건너는 중"으로 통합
  ```

#### 5. 플랜별 배달 시간 재조정
- **현상:** 현재 모든 플랜 168h(7일) 고정. 3:1에서 Premium 3일 대기 = ToThereOn 1일. 세계관상 "편지 하나 보내고 답장 받는데 ToThereOn 기준 1일"은 너무 빠름
- **적용 방법:**
  ```
  Free:    7일 대기 (168h) → ToThereOn ~2.3일 소요 — 적절
  Basic:   5일 대기 (120h) → ToThereOn ~1.7일 소요 — 적절
  Premium: 3일 대기 (72h)  → ToThereOn 1일 소요 — 경계선
  
  ⚠️ Premium 3일은 유지하되, 상태 애니메이션을 "특급 배달" 느낌으로 차별화
  ```

#### 6. Day Arc 챕터 체감 속도 변화
- **현상:** THE CROSSING(day 1) = 이전 7일→이제 3일. MAPPING THE WORLD(day 2-6) = 이전 14-42일→이제 6-18일. 유저가 체감하는 스토리 진행이 2배 이상 빨라짐
- **영향:** 리텐션에 긍정적 (빠르게 성장하는 느낌) but 조기 콘텐츠 고갈 위험
- **적용 방법:** Day Arc 일수 자체는 ToThereOn일 기준이므로 코드 수정 불필요. 다만 Phase 1에서 Day 100+ 콘텐츠(A LEGEND) 깊이를 더 빨리 준비해야 함
  ```
  Day 100 도달 시점: 700일(~2년) → 300일(~10개월)
  → STAGE 6-7, The Still Pool 등 심화 콘텐츠 우선순위 상향 필요
  ```

#### 7. 피드 생성 빈도와 세계 시간 정합성
- **현상:** Free 주1회 피드 = 7일에 1번 = ToThereOn 2.3일에 1번. 아이가 2일 넘게 아무 소식 없으면 "거기서 뭐하고 있는 거지?" 감각
- **적용 방법:**
  ```
  기존 (7:1 기준):
  Free:    주 1회 = ToThereOn 1일에 1번 — 딱 맞음
  Basic:   주 2회 = ToThereOn 2일에 1번
  Premium: 격일   = ToThereOn ~3일에 2번

  변경 후 (3:1 기준):
  Free:    주 1회 = ToThereOn 2.3일에 1번 — 약간 느슨하지만 수용 가능
  Basic:   주 2회 = ToThereOn ~1.2일에 1번 — 적절
  Premium: 격일   = ToThereOn ~0.7일에 1번 — 오히려 하루에 2번 가까움
  
  ⚠️ Premium 격일 피드가 ToThereOn 하루 안에 2개 생길 수 있음
  → 피드 생성 시 "같은 ToThereOn Day에 이미 피드 있으면 스킵" 로직 강화 필요
  ```

### 🟢 Minor (인지하고 넘어갈 수 있음)

#### 8. 학습 성장(STAGE) 체감 가속
- **현상:** STAGE 5(글씨 완성) 도달이 84-91일→36-39일로 단축. 유저가 "한 달 만에 아이가 완전한 편지를 쓴다"는 감동을 더 빨리 경험
- **평가:** 리텐션에 **긍정적**. 첫 달 내 핵심 감동 포인트(글씨 완성) 경험 → 유료 전환 동기 강화
- **적용:** 코드 수정 불필요 (STAGE 기준이 ToThereOn일이므로 자동 반영)

#### 9. 기존 유저 데이터 마이그레이션
- **현상:** 이미 등록된 펫이 있다면, 기존 7:1로 계산된 currentDay가 3:1로 재계산 시 급격히 증가
  - 예: 현실 21일 경과 펫 → 기존 Day 3 → 변경 후 Day 7 (존이 crystal_meadow → eternity_forest로 점프)
- **적용 방법:** 출시 전이므로 테스트 데이터만 존재. **DB ratio 값만 변경하면 자동 반영** (time-engine이 매번 실시간 계산하므로)
- **출시 후 변경 시엔:** 마이그레이션 스크립트 + 이전 피드와의 연속성 처리 필요

#### 10. 세계관 설명 문구 업데이트
- **현상:** 온보딩, 시스템 프롬프트, UI 곳곳에 "7 Earth days = 1 ToThereOn day" 하드코딩된 텍스트 존재
- **적용 방법:** 검색 & 치환
  ```
  수정 대상 문구:
  - "Every seven days you live" → "Every three days you live"
  - "1 day here equals 7 days on Earth" → "1 day here equals 3 days on Earth"  
  - immersion_rules: "1 ToThereOn day = 7 Earth days" → "1 ToThereOn day = 3 Earth days"
  - 체크리스트 프롬프트 내 모든 시간 비율 언급
  ```

## 적용 순서 (권장)

```
Step 1: DB 값 변경
  → admin_configs.time_rules.ratio: 7 → 3
  → worldview-constants.ts RATIO: 7 → 3

Step 2: 텍스트 일괄 치환
  → 시스템 프롬프트, 체크리스트, 세계관 설명 문구

Step 3: 검수 AI 예외 규칙 추가 (Critical #2)
  → Haiku가 배달 시간을 오판하지 않도록

Step 4: 편지 상태 파이프라인 재설계 (Critical #1, #3, Major #4)
  → 상태 전환 타이밍 + letter_response 트리거 조건 + 시점 명시

Step 5: 피드 빈도 정합성 검증 (Major #7)
  → Premium 격일 피드의 중복 방지 로직

Step 6: lazy delivery 플랜별 동적값 (Major #5)
  → Free 168h / Basic 120h / Premium 72h
```
